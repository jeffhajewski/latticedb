One local file. Three query modes. Zero platform theater.

LatticeDB is an embedded property-graph database for connected, semantic, and textual data. Keep graph traversal, vector similarity, BM25 full-text search, and durable change streams in the same engine and the same file.

Install without hunting through docs.

Four entry paths, one obvious first step. The goal of this redesign is to make the shortest route visible immediately while still exposing the deeper workflows for repo-local development and bindings.

fast path Pick the interface you actually want to use.

The current repo already has clear commands. This layout surfaces them early, keeps the friction low, and then adds just enough context to prevent a dead end.

  • CLI for importing data, running queries, and inspecting files.
  • Python for local apps, scripts, and notebooks.
  • TypeScript for Node.js tooling and services.
  • Go for cgo-backed native workflows.

Published Python wheels and TypeScript tarballs are expected to bundle liblattice on supported platforms. Go uses an installed-prefix or repo-local shared-library workflow.

CLI install 
curl -fsSL https://raw.githubusercontent.com/jeffhajewski/latticedb/main/dist/install.sh | bash

Best starting point if you want the shortest path to a working database file.

Python install 
pip install latticedb

For local applications, notebooks, and scripts. Supported wheels are expected to bundle the native library.

TypeScript / Node.js install 
npm install @hajewski/latticedb

For Node services and tooling. Supported package tarballs are expected to bundle the native library.

Go workflow 
zig build install --prefix /tmp/lattice-install
export PKG_CONFIG_PATH=/tmp/lattice-install/lib/pkgconfig
# macOS
export DYLD_LIBRARY_PATH=/tmp/lattice-install/lib
# Linux
export LD_LIBRARY_PATH=/tmp/lattice-install/lib
cd bindings/go && go test ./...

Go is cgo-backed today. For repo-local development, build the shared library and use go test -tags repolocal ./....

core story One engine for three ways of asking for data.

Traversal, vector similarity, and BM25 full-text search belong in the same mental model. The site should make that visible early instead of treating them as scattered subfeatures.

Five-minute path.

The current getting-started guide is good. This experiment compresses it into a single visual strip so a visitor can tell whether the product is easy before scrolling farther.

step 01 Create a database. 
$LATTICE create /tmp/lattice-quickstart.lattice

Either install the CLI or build locally with zig build and point $LATTICE at ./zig-out/bin/lattice.

step 02 Import the sample graph. 
$LATTICE import /tmp/lattice-quickstart.lattice \
  --file=examples/cli/people_graph.json

Start from a small author/document graph instead of a blank database.

step 03 Inspect counts. 
$LATTICE count /tmp/lattice-quickstart.lattice

Quick feedback that the file exists, the import worked, and the graph is queryable.

step 04 Run a query file. 
$LATTICE exec /tmp/lattice-quickstart.lattice \
  --file=examples/cli/authored_docs.cypher

Then verify file checksums with $LATTICE check /tmp/lattice-quickstart.lattice.

Show the query model, not just the promise.

The fastest way to explain LatticeDB is to show one snippet that crosses vector, text, and graph traversal in the same query path.

example shape One query layer.

This is the part that should anchor the page: not marketing abstraction, but one concise example that proves why these primitives belong together.

  • Match chunks and walk to documents and authors.
  • Filter semantically with vector distance.
  • Add lexical precision with BM25 full-text conditions.
MATCH (chunk:Chunk)-[:PART_OF]->(doc:Document)-[:AUTHORED_BY]->(author:Person)
WHERE chunk.embedding <=> $query_vector < 0.3
  AND doc.content @@ "neural networks"
RETURN doc.title, chunk.text, author.name
ORDER BY chunk.embedding <=> $query_vector
LIMIT 10

That combination is the thesis of the product, so the redesign keeps it physically near the install path and quickstart.

operating notes What a serious reader wants to know early.
  • Embedded, single-file, local-first database.
  • Single-writer model for one owning process.
  • WAL-backed durability for graph writes and streams.
  • Bindings for Python, TypeScript, and Go on top of the stable C ABI.
  • Examples include graph/vector/text retrieval demos.
